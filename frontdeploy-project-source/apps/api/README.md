# Frontdeploy API

Read-only backend API for Frontdeploy. The service is intentionally
small, mock-first, and safe to run locally while provider integrations and data
storage are developed.

## Current Features

- Fastify API server.
- Strict TypeScript build.
- CORS allowlist for Chrome extension and local development origins.
- Health endpoint.
- Deterministic mock intelligence endpoint for Solana-style addresses.
- Optional PostgreSQL/Supabase storage for wallet labels, wallet profiles, and token risk cache.
- Backend-side provider adapters for GMGN, Solana Tracker, GoPlus, Helius, Birdeye, Jupiter, and DexScreener.
- Optional local/mock AI summary endpoint.
- Cloud label sync endpoints are disabled by default until auth is configured.
- Runtime dependency audit in `npm run verify`.
- Endpoint tests with Node's built-in test runner.

## Provider Setup

Provider keys cannot be generated by this repo. Create them in each provider
dashboard, then place them in `apps/api/.env`.

- Helius: `HELIUS_API_KEY` for DAS/RPC wallet and token metadata.
- Birdeye: `BIRDEYE_API_KEY` for token security data.
- Jupiter: `JUPITER_API_KEY` for price/liquidity data when your Jupiter plan requires it.
- GMGN: `GMGN_API_KEY` for read-only token info, security, pool, and holder/trader analytics through `gmgn-cli`.
- Solana Tracker: `SOLANA_TRACKER_API_KEY` for token safety/rugcheck scores, freeze/mint authority flags, insider risk, and liquidity depth.
- GoPlus: `GOPLUS_API_KEY` for Solana token security checks when you want a Birdeye alternative.
- DexScreener: no API key required for public Solana token pair data.

GMGN is integrated only for read-only token intelligence. Do not configure
`GMGN_PRIVATE_KEY`; this project does not support GMGN swap, order, copy
trading, or wallet-management commands.

Install GMGN CLI if you want GMGN enrichment:

```bash
npm install -g gmgn-cli
```

GMGN publishes a demo key for connection testing in their docs, but production
usage should use your own key from `https://gmgn.ai/ai`.

Check configured providers without exposing secret values:

```bash
curl "http://127.0.0.1:8787/v1/providers/status"
```

On Windows PowerShell, use the setup helper to write keys without echoing them:

```powershell
.\scripts\set-api-keys.ps1
```

Official docs:

- Helius DAS: https://www.helius.dev/docs/das-api
- Birdeye token security: https://docs.birdeye.so/reference/get-defi-token_security
- Jupiter Price API: https://dev.jup.ag/docs/price/v3
- DexScreener API: https://docs.dexscreener.com/api/reference
- GMGN Agent API and CLI: https://docs.gmgn.ai/index/gmgn-agent-api
- Solana Tracker token safety: https://docs.solanatracker.io/guides/token-safety
- GoPlus Solana token security: https://docs.gopluslabs.io/reference/solanatokensecurityusingget

Do not commit `.env`, provider keys, private keys, seed phrases, browser
cookies, or exchange credentials.

## Endpoints

### `GET /health`

Returns service status.

### `GET /v1/providers/status`

Returns provider configuration status without returning secret values.

### `GET /v1/intelligence/:address`

Returns mock intelligence for a Solana-style Base58 address.
When provider keys are configured, the backend enriches the response with
read-only GMGN, Solana Tracker, GoPlus, Helius, Birdeye, Jupiter, and
DexScreener data. DexScreener is public and does not require a key. Birdeye is
optional; Solana Tracker and GoPlus can cover token safety/risk if Birdeye is
not configured.
When `DATABASE_URL` is configured, the backend stores the generated wallet
profile or token risk snapshot in PostgreSQL/Supabase for caching.

Query params:

- `kind=auto`
- `kind=wallet`
- `kind=token`

Example:

```bash
curl "http://127.0.0.1:8787/v1/intelligence/7YttLkHDo5e7FbrU6mJK7L5zXWbhP2hUv3Y1qVhU6nNq?kind=wallet"
```

### `POST /v1/reputation/developer`

Audits project/developer trust evidence for a token. It checks whether the
claimed contract address appears on the supplied website and GitHub README,
reads public GitHub repository metadata, scores market cap thresholds, and
records X/social narrative links as manual evidence.

Example:

```bash
curl -X POST "http://127.0.0.1:8787/v1/reputation/developer" \
  -H "Content-Type: application/json" \
  -d "{\"tokenAddress\":\"So11111111111111111111111111111111111111112\",\"websiteUrl\":\"https://example.com\",\"githubRepoUrl\":\"https://github.com/owner/repo\",\"marketCapUsd\":10000,\"narrative\":\"Coinbase replied to the launch post\"}"
```

### `GET /v1/labels/:address`

Returns a cloud wallet label when `CLOUD_LABEL_SYNC_ENABLED=true`,
`DATABASE_URL` is configured, and a future authenticated user id is supplied in
`x-axiom-user-id`.

### `PUT /v1/labels/:address`

Saves a cloud wallet label under the same disabled-by-default guard. This is
prepared for future auth and should not be exposed publicly without a real auth
layer.

### `POST /v1/ai/summary`

Returns a safe mock AI summary for wallet or token facts. This endpoint does not
call paid AI APIs yet and must not receive private keys, seed phrases, cookies,
or exchange credentials.

Example:

```bash
curl -X POST "http://127.0.0.1:8787/v1/ai/summary" \
  -H "Content-Type: application/json" \
  -d "{\"address\":\"7YttLkHDo5e7FbrU6mJK7L5zXWbhP2hUv3Y1qVhU6nNq\",\"kind\":\"wallet\",\"riskScore\":72,\"facts\":[\"High fresh-wallet activity\"]}"
```

## Supabase Setup

The backend needs a Postgres connection string, not a Supabase personal access
token. Use the Supabase dashboard to copy the project `DATABASE_URL` URI, then
run:

```powershell
.\scripts\setup-supabase.ps1
```

Optional: set `SUPABASE_ACCESS_TOKEN` in your local shell before running the
script if you want it to list visible Supabase projects. The token is not saved
to `.env` and is not required for runtime.

The script writes `apps/api/.env`, sets `DATABASE_SSL=true`, and applies:

```text
apps/api/supabase/migrations/001_initial_intelligence_schema.sql
```

## Safety Boundaries

The backend must remain read-only. It must not connect wallets, request private
keys, request seed phrases, execute trades, copy trade, or sign transactions.
Provider keys belong only in backend environment variables and should never be
stored in the extension.
